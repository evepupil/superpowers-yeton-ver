# QA 风险 Review

## 核心定位

基于代码提交区间做测试视角的白盒风险评审。重点不是代码风格，而是隐藏缺陷、测试风险、业务一致性风险和必须修复项。

材料不足时不要开始 review。

## 必需输入

开始前必须确认：

- 仓库地址或本地仓库路径
- 目标分支
- 起始提交 ID
- 结束提交 ID

如果任一项缺失，先向用户索要。不要只凭当前工作目录或最近提交猜测 review 范围。

## 仓库处理

若用户提供远程仓库地址：

1. 克隆到用户确认的临时目录或当前工作目录下的工作区
2. fetch 最新分支和提交
3. checkout 目标分支
4. 确认起始提交和结束提交都存在

若用户提供本地仓库路径：

1. 进入该仓库
2. 检查当前工作树状态
3. fetch 或提示用户是否需要更新远程
4. checkout 目标分支
5. 确认提交区间存在

不要修改业务代码。不要自动合并、reset、rebase、删除文件或清理用户工作区。

## Diff 范围

使用用户指定的提交区间进行 review，例如：

```bash
git diff <起始提交>..<结束提交>
git log --oneline <起始提交>..<结束提交>
```

如起始提交是否包含在范围内存在歧义，向用户确认。默认 review “起始提交之后到结束提交为止”的改动。

## Review 重点

优先检查：

- 空值、未定义、空集合、空字符串处理
- 边界条件、状态边界、时间边界、金额/数量边界
- 权限绕过、越权访问、角色判断遗漏
- 事务一致性、部分成功、失败回滚
- 并发、重复提交、幂等、竞态条件
- 错误处理、异常吞掉、用户不可见失败
- 日志缺失、敏感信息日志、排障信息不足
- 数据一致性、缓存一致性、读写顺序
- 配置、环境变量、默认值风险
- 单测、集成测试、回归测试缺失

## 输出产物

默认输出 `04-白盒review报告.md`。

推荐结构：

```markdown
# 白盒 Review 报告

## 1. Review 范围
- 仓库：
- 分支：
- 起始提交：
- 结束提交：
- 提交列表：

## 2. 总体结论

## 3. 必须修复问题
| 等级 | 文件/位置 | 问题 | 影响 | 建议 |

## 4. 风险项
| 等级 | 文件/位置 | 风险 | 触发条件 | 建议测试 |

## 5. 测试补充建议
| 业务场景 | 建议补充测试点 | 原因 |

## 6. 单测/自动化缺口

## 7. 未确认问题
```

## 风险等级

- P0：可能导致核心流程不可用、严重数据错误、严重权限/安全问题、资金或关键业务损失
- P1：高概率影响主要流程、造成明显错误结果、阻塞测试或验收
- P2：边界缺陷、可恢复异常、体验或可维护性风险
- P3：低风险建议，不影响本次交付

输出时把 P0/P1 放前面。没有发现问题时，也要说明 review 范围和残余风险。

## 严禁事项

- 不要只做风格评论
- 不要脱离 diff 范围泛泛建议
- 不要修改用户代码
- 不要把不确定推断写成确定缺陷
- 不要忽略缺失测试的风险
